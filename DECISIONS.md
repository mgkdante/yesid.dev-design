# P5.4 DECISIONS - public distribution

Slice `P5.4` · 2026-07-16. This decision supersedes only the local-only distribution assumptions in D2 and D8. Their extraction and first-adopter history remains accurate.

## D13 - Distribution is pinned-tag vendoring from the public repository

**Status:** accepted.

**Context.** The repository is public at `https://github.com/mgkdante/yesid.dev-design`, so consumers no longer need a sibling checkout or registry credentials. A Git dependency is not a package-level solution for this monorepo: package managers resolve the repository root and cannot select `packages/tokens`, `packages/motion`, `packages/gates`, and `packages/ui` as independent workspace packages. npm publication would solve package selection, but it adds registry ownership, release automation, provenance, and support work that is not needed yet.

**Decision.** Consumers vendor the requested packages from a public GitHub tag. The tag is exact and never floats. `tools/adopt.ts` shallow-clones that tag over public HTTPS, copies the selected packages with the Transit exclusion rules, rewrites internal `workspace:*` links to sibling `file:` links, copies the MIT notice, and writes `manifest.json` with the tag, commit, package set, and deterministic tree hash. `--check` proves the committed snapshot has not been edited. `--source` uses a local checkout for development without network access; it does not prove that the supplied tag points at that checkout.

Consumers may copy the script into their repo or run an equivalent sync script. They commit the vendored snapshot and manifest. A brand bump is explicit: choose a new tag, rerun the tool, review the generated diff, regenerate consumer-owned token outputs, and merge that bump deliberately. Vendored package code is never patched in the consumer.

Transit remains the reference implementation. Adding public remote-fetch mode to Transit's `apps/web/tools/design-sync.ts` is a separate Transit-side change; this repository does not modify Transit during P5.4.

**Consequences.** New products can adopt without credentials. CI can verify a snapshot offline after adoption. The repository tag and manifest become the distribution receipt. Consumers still own the small integration seams around token output paths, application CSS, localized copy, and product-specific gate configuration.

**Deferred option.** npm publishing can be added later as an optional layer. It does not replace exact pins, deliberate bump reviews, or the one-direction flow law.

## D15 - Prospective v0.7.0 source exports and opt-in CSS

**Consequence.** This prospectively supersedes D11's copy-paste-only delivery for tap feedback without rewriting that historical `v0.3.0` evidence. ## D16 - Prospective v0.7.0 UI props describe the rendered contract

**Status:** accepted 2026-07-17.

**Context.** Several extraction-era wrappers exposed broader types than their runtime behavior: Button and Badge used truthy `href` branching while mixing element attributes, fixed-content wrappers advertised replaceable children, Combobox omitted state and callback controls it needed to compose, custom Separator variants inherited bits-ui-only props they did not render, Sheet wrapper props were mostly unnamed, and Toggle Group erased its single/multiple value distinction with a `never` cast while making Item value appear bindable.

**Decision.** Beginning with the prospective `v0.7.0` contract, public props follow the element or state each wrapper actually owns. Button and Badge discriminate on nullish versus string `href`, with the empty string intentionally rendering an anchor. Fixed-content wrappers reject children, and wrapper-owned bits-ui render hooks are not public. Combobox exposes optional bindable nullable `value`, bindable `open`, selected-label/input synchronization, and composed root callbacks. Separator delegates the default branch to bits-ui but gives hazard and gradient exact native-div contracts with outer ref, `data-slot`, and `maxWidth` behavior. Sheet exports named props for all public parts and gives Content caller-owned `closeLabel` copy while retaining ownership of nested portal children. Toggle Group preserves the single/string and multiple/string-array union with branch-specific bindings; Item value is required input-only while ref remains bindable.

**Consequence.** Valid consumer behavior and package styling stay compatible, while impossible or misleading prop combinations become type errors and the previously missing state wiring becomes observable. Historical parity notes remain evidence of the extraction snapshots; they do not define this prospective API. Consumer adoption and visual verification remain separate changes, and this decision makes no consumer-repository edits.

## D17 - Prospective v0.7.0 UI configuration is single-assignment per ESM module graph

**Status:** accepted 2026-07-17.

**Context.** The UI package's class merger is module-scoped, but the initial configuration hook allowed later calls to replace earlier vocabulary. That made import and render order capable of changing class resolution, made an empty call behave like an implicit reset, and left consumers without an explicit result for safe repeated initialization. A root layout instance script is also the wrong lifecycle boundary for state shared by package imports across SvelteKit rendering.

**Decision.** Beginning with the prospective `v0.7.0` contract, `configureUi` accepts one semantic configuration per loaded ESM module graph. The first explicit call in an unlocked graph stores a normalized copy and returns `'initialized'`. Omitted configuration, omitted vocabulary fields, and empty arrays are the same zero configuration; duplicate entries are removed and ordering is ignored. A semantically equivalent repeat returns `'unchanged'`. A conflicting repeat throws and preserves the first merger. The first `cn` use locks the zero-configuration default when no explicit call has happened, and there is no reset API. The exported `ConfigureUiResult = 'initialized' | 'unchanged'` union is observable prospective public API.

Products configure only fixed application vocabulary. Request-, tenant-, user-, locale-, or other runtime-scoped configuration is forbidden. SvelteKit consumers define one shared initializer and import it from both `hooks.client.ts` through `ClientInit` and `hooks.server.ts` through `ServerInit`; they do not initialize from a root layout instance script.

**Consequence.** Initialization order is deterministic inside a module graph and repeated boot calls are safe only when semantically equivalent. The browser bundle, each SSR worker, separately bundled package copies, and any graph recreated by HMR have independent or recreated state and run their own initialization. This is module isolation, not cross-worker or cross-bundle global state.

**Deferred.** Provider/context configuration remains outside `v0.7.0`. It may be reconsidered only if a future product requires a deliberately runtime-scoped contract; it is not a route around the fixed boot configuration decided here.

## D18 - U2 token generation has one upstream artifact and downstream path ownership

**Status:** accepted 2026-07-18.

**Context.** The extraction-era token build mixed deterministic generation with repository I/O. It knew gallery, motion, and root-document paths, wrote a byte-identical gallery CSS mirror, and let upstream package tests reach back into consumers. That violated the repository's one-direction flow law and made path behavior depend on the host checkout.

**Decision.** `packages/tokens/src/build.ts` is a private, pure logical engine: content in, named outputs out. It contains no filesystem or consumer knowledge. `tools/build-tokens.ts` is the repository adapter and is the only owner of physical output paths, sentinel replacement, public UI inventory discovery, and writes. `packages/tokens/tokens.css` is the single committed CSS artifact; gallery imports the public `@yesid/tokens/tokens.css` subpath. Motion TypeScript, gallery's generated theme region, and `DESIGN.md` remain committed downstream artifacts. Raw generated bytes must match on Ubuntu and Windows.

**Consequence.** This prospectively supersedes D15's two-copy CSS sentence without rewriting its historical context. The public package export stays stable, repository build machinery remains private, generators are reusable without a checkout, and consumers no longer leak into the token domain.

## D19 - U3 gate engines are policy-neutral

**Status:** accepted 2026-07-18.

**Context.** The extraction package published yesid.dev and Transit preset subpaths, concrete contrast tables, forbidden-style rules, brand hexes, and an app-named color-mix default. That made consumer policy part of a supposedly reusable engine package and encouraged apps to share decisions that belong to their own visual and accessibility contracts.

**Decision.** Beginning with the prospective `v0.7.0` contract, `@yesid/gates` has one public subpath, its root. Production source contains pure parameterized engines, their types, and only app-neutral defaults. It contains no app-named preset modules, symbols, or policy tables. `BrandHexConfig.hexes` is required so a consumer cannot silently inherit another product's palette. The Gallery owns the small policy it exercises. Neutral frozen policy lives only under the package test boundary and is not exported or vendored. Each external consumer owns one local policy module and an exact contract test, then passes that policy to the shared engines.

**Consequence.** This prospectively supersedes only D15's promise that the two gate preset subpaths remain unchanged. Existing exact pins continue to protect deployed consumers until their deliberate `v0.7.0` bump. A consumer with tokens beyond the neutral color-mix defaults must provide its complete token list explicitly. Engine fixes remain reusable upstream; product policy changes stay isolated to the product that owns them.

## D20 - U4 adoption is a verified acquisition and atomic install protocol

**Status:** accepted 2026-07-18.

**Context.** The first public adoption tool shallow-cloned a tag, recorded only `HEAD`, lived in one consumer-copied file, and replaced the destination without a durable recovery protocol. That was adequate distribution scaffolding, but it could not distinguish an annotated tag object from its peeled commit, bind bytes to an immutable Release asset, prove the executing tool and exclusion policy, or recover deterministically from interruption between backup and install.

**Decision.** Schema 2 has one deterministic trust payload: fixed repository identity; acquisition mode; annotated tag object and peeled commit; immutable Release asset name, byte size, and SHA-256 digest when in production mode; canonical package closure; exclusion-policy digest; complete self-vendored tool digest; and a length-framed full tree hash. It contains no timestamp. Production acquisition accepts only the canonical bounded POSIX ustar asset from the exact published immutable GitHub Release and cross-checks its embedded receipt. Local archive and clean annotated-worktree snapshots are explicit development modes and cannot claim Release-asset provenance.

The tool is split by domain into acquisition, contract, payload, transaction, and thin CLI modules. Installation uses a destination lock, sibling stage, durable prior-version backup, pre- and post-swap verification, exact-manifest no-op detection, and deterministic recovery. Operational exits `0` through `7` distinguish success, internal failure, usage, precondition, lock contention, check failure, transaction failure with rollback, and recovery-required state. The complete tool bundle is part of every adopted payload, so an offline `--check` verifies the same policy that produced the receipt.

**Consequence.** This prospectively supersedes D13's shallow-clone, schema-1, single-file, and unverified local-source details without rewriting its historical distribution decision. Consumer bumps remain exact and reviewable, but a valid receipt now binds source identity, package closure, policy, tool, and installed bytes. A failed transaction preserves one complete accepted version or stops with explicit recovery evidence; it never silently presents a partial adoption as successful.
